\section{Technologies retenues}

\subsection{Traitement distribué : Ray}

\gls{ray} distribue des workflows Python sur \gls{gpu}.
Contrairement à Spark, conçu pour les données structurées, Ray est optimisé pour
l'inférence de modèles, ce qui en fait le choix naturel pour ce pipeline.

\subsection{Observabilité : à définir}

Le pipeline expose des métriques en temps réel : utilisation \gls{gpu}, mémoire,
latence d'inférence et progression du batch. Prometheus et Grafana sont les candidats
naturels dans un environnement \gls{kubernetes} ; le choix sera justifié dans le rapport.

\subsection{Stockage objet : MinIO}

\gls{minio} est la solution de stockage objet retenue. Compatible \gls{s3},
il est déjà déployé sur l'infrastructure de la HEIG-VD et partagé par d'autres projets du département.

\subsection{Persistance des résultats : Parquet sur S3}

Les annotations et métadonnées produites par \gls{sam3} sont stockées en fichiers
\gls{parquet} sur le bucket \gls{s3}. Ce format columaire est nativement supporté
par Ray Data, évite l'overhead d'une base de données et s'intègre directement
dans le pipeline batch.

\subsection{Orchestration : Kubernetes}

Le pipeline est déployé sur le cluster \gls{kubernetes} de la HEIG-VD sous forme
de manifestes (Jobs Ray, Deployments). \gls{rancher} assure l'administration
et le monitoring du cluster.

\subsection{Modèle : SAM3}

\gls{sam3} (Segment Anything Model v3) est utilisé en inférence uniquement.
